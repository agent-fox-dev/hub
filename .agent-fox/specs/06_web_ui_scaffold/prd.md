# Web UI Scaffold

## Background

af-hub requires a web UI so that operators and users can interact with the platform through a browser. The UI will eventually provide dashboards, settings management, and operational views. Rather than building all pages at once, this scaffold spec establishes the project structure and toolchain so that future specs can add functional pages incrementally, without needing to revisit foundational tooling decisions.

**Toolchain rationale:**
- **Vite** — fast dev server with first-class TypeScript support and straightforward proxy configuration.
- **React** — team familiarity and a large ecosystem of compatible libraries.
- **Tailwind CSS** — utility-first styling that enables rapid prototyping and consistent design without a heavy CSS architecture.
- **shadcn/ui + Radix UI** — accessible, composable components copied directly into the source tree, giving the team full control over component internals without being locked to an npm release cycle.
- **React Router** — the standard choice for client-side routing in React applications.
- **TanStack Query** — powerful async state management that will be used for API integration in future specs; configured here so all future pages share a consistent data-fetching foundation.
- **TypeScript** — project-wide type safety across all frontend code.

## Intent

Set up the web frontend project structure for af-hub. This delivers the toolchain, dev proxy, build targets, and a minimal hello-world route — ready for future functional pages.

This is minimally dependent on the backend: only the Vite dev proxy configuration references the backend's health probe paths.

## Goals

- Initialize the `web/` project at the repo root with its own `package.json`.
- Set up Vite + React + TypeScript + Tailwind CSS + shadcn/ui (full CSS variable theming convention).
- Configure the Vite dev server to listen on port **5173** (Vite default).
- Configure the Vite dev proxy to forward `/api`, `/healthz`, and `/readyz` to the Go backend (default `localhost:8080`) with verbatim path forwarding and `changeOrigin: true`.
- Configure a `@/` path alias (mapping to `src/`) in both `tsconfig.json` and `vite.config.ts`.
- Commit `components.json` (generated by `npx shadcn-ui init`) to the repository.
- Add `make web-dev`, `make web-build`, and `make web-lint` Makefile targets.
- Ship a single hello-world route at `/` and a minimal 404 catch-all route.
- `npm run dev` starts the Vite dev server on port 5173 with hot reload.
- `npm run build` produces a production build to `web/dist/`.
- `npm run lint` runs ESLint + TypeScript type checking.
- Commit a `.nvmrc` pinning Node.js 22 LTS (placed at `web/.nvmrc`) and a `package-lock.json` lockfile.
- Add `web/dist/` and `web/node_modules/` to the root `.gitignore`.

## Non-goals

- Functional pages (login, dashboard, settings).
- Authentication flows in the web UI.
- API client library or data fetching.
- Server-side rendering.
- Production serving of `web/dist/` (go:embed, CDN, separate static host, etc.).
- CI pipeline integration (deferred to a future spec — the new Makefile targets exist but are not yet wired into any CI workflow).
- Unit or component testing infrastructure (e.g., Vitest, React Testing Library) — deferred to a future spec.
- Dark mode class-switching mechanism — `tailwind.config.ts` uses `darkMode: 'class'` per shadcn/ui convention, but no toggle or class-application mechanism is provided in this spec. The UI will render in light mode by default. Dark mode wiring is deferred to a future spec.

## Dependencies

| Spec | Relationship | Notes |
|------|-------------|-------|
| `server_foundation` | Runtime dependency (dev proxy) | The Vite dev proxy forwards `/healthz` and `/readyz` to the Go backend, which is defined and owned by `server_foundation`. The proxy will fail silently if the backend is not running; this is expected during frontend-only development. |

## Functional Requirements

### Environment pinning

- A `.nvmrc` file must be committed at **`web/.nvmrc`** specifying Node.js **22** (LTS). This is the single canonical location; no repo-root `.nvmrc` is created by this spec.
- The `package-lock.json` lockfile must be committed and kept up-to-date to ensure reproducible installs in CI and across developer environments.
- Package versions in `package.json` use semver ranges (`^`); the lockfile is the source of truth for exact resolved versions.

### Repository hygiene

- `web/dist/` and `web/node_modules/` must be added to the **root-level `.gitignore`** (not a `web/.gitignore`) so that build artifacts and installed packages are never committed.
- No other web-specific ignore rules are required beyond these two entries.

### Project initialization

- Create the `web/` directory with the following top-level files:
  - `package.json`
  - `tsconfig.json`
  - `tsconfig.node.json`
  - `vite.config.ts`
  - `tailwind.config.ts`
  - `postcss.config.js`
  - `eslint.config.js`
  - `components.json` (generated by `npx shadcn-ui init`; must be committed)
  - `index.html` (Vite entry point at the `web/` root, not inside `src/`)
  - `.nvmrc` (Node 22 LTS)
  - `package-lock.json` (committed lockfile)
- The `src/` directory structure must follow the standard Vite scaffold layout:
  ```
  web/
  ├── index.html
  ├── .nvmrc
  ├── package.json
  ├── package-lock.json
  ├── tsconfig.json
  ├── tsconfig.node.json
  ├── vite.config.ts
  ├── tailwind.config.ts
  ├── postcss.config.js
  ├── eslint.config.js
  ├── components.json
  └── src/
      ├── main.tsx          # React entry point; mounts <App /> into #root
      ├── App.tsx           # Root component; sets up Router + QueryClientProvider
      ├── globals.css       # Tailwind directives + shadcn/ui CSS variable declarations
      ├── vite-env.d.ts     # Vite client types + __APP_VERSION__ ambient declaration
      ├── lib/
      │   └── utils.ts      # cn() helper generated by npx shadcn-ui init
      ├── routes/           # One file per route; index route lives here
      │   ├── index.tsx     # Hello-world route component rendered at /
      │   └── not-found.tsx # 404 catch-all route component
      ├── components/       # Shared/reusable components (shadcn copies land here)
      │   └── ui/           # shadcn/ui component copies (e.g., button.tsx)
  ```
- Install dependencies (semver ranges):
  - **Runtime:** `react`, `react-dom`, `react-router-dom`, `@tanstack/react-query`
  - **Dev:** `typescript`, `vite`, `@vitejs/plugin-react`, `tailwindcss`, `postcss`, `autoprefixer`, `eslint` (v9+), `@typescript-eslint/eslint-plugin`, `@typescript-eslint/parser`, `eslint-plugin-react`, `eslint-plugin-react-hooks`
- Initialize shadcn/ui with components copied into the source tree (not installed as an npm dependency). **At minimum, the `Button` component must be copied in** (via `npx shadcn-ui add button` or equivalent) to validate the shadcn/ui setup end-to-end. The Button component must be used or rendered in the hello-world route.

### TypeScript configuration

- `tsconfig.json` must enable the following compiler options at minimum:
  ```json
  {
    "compilerOptions": {
      "strict": true,
      "noUncheckedIndexedAccess": true,
      "moduleResolution": "bundler",
      "target": "ES2022",
      "lib": ["ES2022", "DOM", "DOM.Iterable"],
      "jsx": "react-jsx",
      "module": "ESNext",
      "skipLibCheck": true,
      "noEmit": true,
      "baseUrl": ".",
      "paths": {
        "@/*": ["./src/*"]
      }
    }
  }
  ```
- A `tsconfig.node.json` must be committed at `web/tsconfig.node.json` to cover `vite.config.ts` and other Node-side config files. It must use `moduleResolution: "bundler"` (or `"node"`) and target a Node-compatible module format. Example:
  ```json
  {
    "compilerOptions": {
      "composite": true,
      "skipLibCheck": true,
      "module": "ESNext",
      "moduleResolution": "bundler",
      "allowSyntheticDefaultImports": true
    },
    "include": ["vite.config.ts"]
  }
  ```
- The `@/` alias maps to `src/` and must be used by all shadcn/ui component imports (e.g., `import { Button } from "@/components/ui/button"`).
- These settings apply to all TypeScript files under `web/src/`.

### Path alias

- A `@/` path alias must be configured in **both** `tsconfig.json` (via `compilerOptions.paths`) and `vite.config.ts` (via `resolve.alias`), mapping `@/` to the `src/` directory.
- Example `vite.config.ts` alias configuration:
  ```ts
  import path from "path";

  resolve: {
    alias: {
      "@": path.resolve(__dirname, "./src"),
    },
  },
  ```
- This alias is required for the shadcn/ui copied components to resolve their own imports correctly.

### ESLint configuration

- ESLint must use the **flat config format** (`eslint.config.js`) compatible with **ESLint 9+**.
- The config must include TypeScript-aware linting via `@typescript-eslint/eslint-plugin` and `@typescript-eslint/parser`.
- React-specific rules via `eslint-plugin-react` and `eslint-plugin-react-hooks` **must** be included (these are required dependencies, not optional).
- Because this project uses the React 17+ automatic JSX transform (`jsx: "react-jsx"` in `tsconfig.json`), the `react/react-in-jsx-scope` rule **must be explicitly disabled** in `eslint.config.js` to avoid false-positive lint errors. Example:
  ```js
  rules: {
    "react/react-in-jsx-scope": "off",
  }
  ```
- The scaffold must produce **zero lint errors** when `npm run lint` is run.
- `npm run lint` in `package.json` must run both ESLint and TypeScript type checking, e.g.:
  ```json
  "lint": "eslint . && tsc --noEmit"
  ```

### Tailwind CSS and shadcn/ui theming

- Tailwind CSS must be configured using the **full shadcn/ui convention**: CSS variable–based colour tokens declared in `src/globals.css` and referenced in `tailwind.config.ts`.
- `src/globals.css` must include:
  - Tailwind's `@tailwind base`, `@tailwind components`, and `@tailwind utilities` directives.
  - The shadcn/ui CSS variable declarations for `:root` and `.dark` (e.g., `--background`, `--foreground`, `--primary`, etc.) as generated by `npx shadcn-ui init`.
- `tailwind.config.ts` must:
  - Set `content` paths to cover all `.tsx`, `.ts`, `.jsx`, and `.js` files under `src/` (e.g., `["./index.html", "./src/**/*.{ts,tsx,js,jsx}"]`).
  - Extend the theme with the shadcn/ui CSS variable colour mappings (e.g., `colors: { background: "hsl(var(--background))", ... }`).
  - Dark mode strategy: `"class"` (shadcn/ui default). No dark mode toggle is wired up in this spec (see Non-goals).
- `src/globals.css` must be imported in `src/main.tsx` to ensure styles are applied globally.
- This configuration ensures the copied `Button` component renders with correct styles out of the box.

### shadcn/ui `components.json` and `src/lib/utils.ts`

- The `components.json` file generated by `npx shadcn-ui init` must be committed to the repository at `web/components.json`.
- This file records the style, theme, alias, and component path configuration used by the shadcn/ui CLI.
- It is required for all future `npx shadcn-ui add <component>` commands to work correctly without re-running `init`.
- `npx shadcn-ui init` also generates `src/lib/utils.ts`, which exports the `cn()` helper (a `clsx` + `tailwind-merge` utility). This file **must** be present and committed. It is imported by all shadcn/ui component copies (including `Button`) and must not be deleted or omitted from the repository. The `src/lib/` directory and `utils.ts` file must appear in the project directory tree as specified in the Project Initialization section.

### Vite dev proxy

- The Vite dev server must listen on port **5173** (the Vite default; no override required unless a conflict is detected).
- The proxy configuration must use **`changeOrigin: true`** for all forwarded paths.
- All proxy paths must use **verbatim path forwarding** (no path rewrites).
- Forward `/api/*` requests to `http://localhost:8080`.
- Forward `/healthz` to `http://localhost:8080`.
- Forward `/readyz` to `http://localhost:8080`.
- Example `vite.config.ts` proxy configuration:
  ```ts
  server: {
    port: 5173,
    proxy: {
      "/api": {
        target: "http://localhost:8080",
        changeOrigin: true,
      },
      "/healthz": {
        target: "http://localhost:8080",
        changeOrigin: true,
      },
      "/readyz": {
        target: "http://localhost:8080",
        changeOrigin: true,
      },
    },
  },
  ```
- The proxy fails silently (connection refused) when the backend is not running; this is expected and acceptable during frontend-only development.

### Build targets

- `make web-dev` — runs `cd web && npm run dev` (starts Vite dev server on port 5173 with hot reload).
- `make web-build` — runs `cd web && npm run build` (produces production build to `web/dist/`).
- `make web-lint` — runs `cd web && npm run lint` (ESLint + TypeScript type checking).
- All three targets must be added to the top-level `Makefile` as `.PHONY` targets alongside any existing phony targets.
- This spec owns the addition of these three `.PHONY` targets to the root `Makefile`. CI pipeline integration is out of scope for this spec.

### Hello-world route

- A single route at `/` is rendered by the component at `src/routes/index.tsx`.
- The component must display:
  - The text **"af-hub"** as a heading.
  - The **application version**, read from `package.json` at build time via Vite's `define` mechanism:
    ```ts
    // vite.config.ts
    define: {
      __APP_VERSION__: JSON.stringify(process.env.npm_package_version),
    }
    ```
    > **Note:** `process.env.npm_package_version` is populated by npm when Vite is invoked via `npm run dev` or `npm run build`. When invoking Vite directly (e.g., `npx vite`), this variable is `undefined` and `__APP_VERSION__` will be injected as the string `"undefined"`. Developers must use the npm scripts defined in `package.json` rather than invoking `vite` directly.
  - An ambient type declaration for `__APP_VERSION__` must be added to `src/vite-env.d.ts` so TypeScript resolves it without error:
    ```ts
    declare const __APP_VERSION__: string;
    ```
  - At least one **shadcn/ui `Button` component** rendered on the page (e.g., a non-functional placeholder button) to validate the shadcn/ui integration. The Button must be imported using the `@/` alias: `import { Button } from "@/components/ui/button"`.
- React Router is configured for client-side routing (e.g., `BrowserRouter` or `createBrowserRouter` in `src/App.tsx`).
- TanStack Query `QueryClientProvider` is set up in `src/App.tsx` (no queries are executed in this spec).

### 404 catch-all route

- A minimal catch-all route must be registered in the React Router configuration to handle any path that does not match a defined route.
- The catch-all route is rendered by the component at `src/routes/not-found.tsx`.
- The component must display a human-readable message (e.g., "404 — Page not found") and a link back to `/`.
- The catch-all route must use React Router's wildcard path (`path="*"` or equivalent when using `createBrowserRouter`).

### Lint

- `npm run lint` runs ESLint (flat config, ESLint 9+) with TypeScript type checking (`tsc --noEmit`).
- The scaffold must produce **zero lint errors or warnings** (warnings treated as errors is recommended but not required for the scaffold).
- The `react/react-in-jsx-scope` rule must be disabled (see ESLint Configuration section).

## Technical Boundaries

- **Language:** TypeScript (`strict: true`, `noUncheckedIndexedAccess: true`, `moduleResolution: bundler`)
- **Framework:** React
- **Build tool:** Vite
- **Dev server port:** 5173 (Vite default)
- **CSS:** Tailwind CSS with shadcn/ui CSS variable theming convention
- **Components:** shadcn/ui + Radix UI (Button component copied in as validation; `components.json` committed; `src/lib/utils.ts` committed)
- **Path alias:** `@/` → `src/` (configured in both `tsconfig.json` and `vite.config.ts`)
- **Routing:** React Router (client-side, `BrowserRouter` or equivalent); includes `/` and `*` (404) routes
- **Data fetching:** TanStack Query (configured via `QueryClientProvider`, not used)
- **Linting:** ESLint 9+ flat config (`eslint.config.js`) + `@typescript-eslint` + `eslint-plugin-react` + `eslint-plugin-react-hooks` (all required); `react/react-in-jsx-scope` disabled
- **Node.js:** 22 LTS (enforced via `web/.nvmrc`)
- **Package management:** npm with committed `package-lock.json`; semver ranges (`^`) in `package.json`
- **Gitignore:** `web/dist/` and `web/node_modules/` added to root `.gitignore`
- **CI:** Not integrated in this spec; deferred to a future spec
- **Testing:** Not included in this spec; deferred to a future spec
- **Dark mode toggle:** Not included in this spec; deferred to a future spec
